A5 · Cấu trúc một issue

Issue là đầu vào của cả chuỗi (HOW → QA → Code → Review → Test). Ai cũng nên hiểu một issue gồm gì để đọc & làm đúng. Issue có 2 phần: body (nội dung) + custom field (thuộc tính trong Project).

Phần body — viết bằng ngôn ngữ nghiệp vụ

MụcNội dung & cách viết
ControlPhạm vi áp dụng tính năng — quyết định cách dev triển khai (xem bảng Control bên dưới)
ReporterNguồn giao việc + link gốc: Trello / Lark / ai báo trực tiếp — để truy ngược khi cần
WhyPhải rất rõ ràng: tại sao cần làm tính năng này, nhu cầu khách là gì, mục đích sử dụng của khách, tại sao tính năng hiện tại chưa đáp ứng. Mục tiêu: đọc xong biết có đáng để làm hay không
WhatLàm như thế nào — theo ngôn ngữ người dùng: mô tả luồng hiện tại, luồng mong muốn, thao tác sử dụng… Gồm: Tóm tắt · Sơ đồ luồng (Mermaid nếu ≥3 bước/có nhánh) · Mô tả chi tiết step-by-step [Ai] → [làm gì] → [hệ thống phản hồi] → [kết quả] · Acceptance Criteria (đo được, 3–8 điều kiện, cover happy/edge/negative)
DEV NOTEDev điền lại, theo góc độ tech để đáp ứng Why/What của business: Dev Why · Dev What; note ngắn gọn HOW nếu đã có định hướng cách làm
Nguyên tắc viếtWhy/What viết bằng ngôn ngữ người dùng cuối — không nhắc tên file/API/function (cái đó để DEV NOTE). Kiểm tra nhanh: bỏ phần DEV NOTE ra, đưa phần còn lại cho người không biết code đọc → họ vẫn phải hiểu được vấn đề & luồng. Tiếng Việt toàn bộ.

Loại issue: Bug · New Feature · Improve · Maintenance · Infra — mỗi loại có cấu trúc mô tả riêng (Bug có Current/Expected/Steps; Feature có Actors/Happy path/Edge case…).

Luồng status chi tiết — ai phụ trách (PIC) bước nào

Khi QA test FAILTask chuyển sang BUGSSS → dev fix → kéo lại về DOING và đi lại từ đầu.

Cổng cs-request — yêu cầu cho cửa hàng

Nơi gửi các yêu cầu cài đặt cho cửa hàng → team sản phẩm (BA) phê duyệt. Người gửi: CSKH, hoặc chính team sản phẩm khi cần truy cập nhà hàng khách để check/debug, bật flag… 2 loại hay gặp:

① Yêu cầu truy cập nhà hàng (superpass)

Superpass = tài khoản dành riêng cho team nội bộ để truy cập nhà hàng khách (check / debug / test). Hỏi chị Trang để được cung cấp.

Đăng nhập bằng superpass vào nhà hàng cần truy cập — hệ thống báo "Không có quyền" (chưa duyệt).
Gửi yêu cầu tại cs-request → loại "Yêu cầu truy cập nhà hàng".
Chờ duyệt → duyệt xong mới có quyền truy cập.

② Yêu cầu cài đặt cửa hàng — phổ biến: bật/tắt flag

CSKH / team SP gửi request bật flagBA phê duyệtHệ thống bật/tắt flag

Theo dõi request mới qua #cs-request trên Slack. (Lưu ý khi BA phê duyệt flag → xem phần B.)

Nhà hàng test → auto-approveVới nhà hàng test của mình: đăng ký với PM (Trang) để thêm vào danh sách phê duyệt tự động → cần bật flag chỉ cần gửi request, tự động duyệt, không phải chờ người duyệt.

D4 · Implement từ issue

  • Label AI REQUEST CODE (cần HUMAN REVIEW HOW DONE) — request Trang/Hiền gắn để tránh spam token
  • Code theo HOW (đúng file path), follow convention, test local theo QA checklist
  • Gồm cả rebase / resolve conflict + resolve AI review trong luồng
HOW = source of truth — phải LUÔN khớp với codeHOW trên issue là bản chuẩn để code và làm reference cho QA viết checklist. Trong lúc làm, nếu đổi cách làm so với HOW ban đầu → cập nhật lại HOW ngay cho khớp với code đã implement. KHÔNG để HOW một đằng, code một nẻo — người sau (QA, reviewer, dev khác) đọc HOW phải hiểu đúng những gì code thực sự làm.

D6 · Quy chuẩn code & chất lượng

Canonical: code-convention/code-quality-rules.md. Vi phạm các rule dưới → reviewer REQUEST CHANGES.

Giới hạn kích thước

File ≤ 300/500 dòng300 dòng NestJS/React/Next · 500 dòng Node/Flutter
Function ≤ 50 dòngcyclomatic complexity ≤ 10
PR ≤ 300/500 dòngfeature/bug 300 dòng · refactor 500 dòng → vượt thì tách
Test ≥ 3 casehappy/edge/error · coverage ≥ 80% (BLOCK nếu <50%)

POS rules (bắt buộc)

  • Tiền lưu integer (đơn vị nhỏ nhất) — TUYỆT ĐỐI không float/double
  • Timezone-aware — server lưu UTC, hiển thị theo timezone nhà hàng (VN/TH UTC+7, SG UTC+8)
  • Query delete/update phải có restaurantId trong filter — chống sửa/xóa nhầm sang NH khác
  • Entity critical (Order/Payment/User/Restaurant/Menu): soft delete, không hard delete
Cấm (REQUEST CHANGES)any không comment · console.log/print() prod · Promise.all() client · git add . · float cho tiền · new Date() trong business logic · hard delete entity critical · hardcode text user-facing (phải i18n) · PII không mask trong log
Trước khi commitLint pass · test pass · type safe (no any) · PII mask · commit message conventional · 1 PR = 1 việc.
